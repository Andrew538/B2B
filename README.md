# B2B Личный кабинет оптовых покупателей (Архитектурный каркас)

### 📌 Описание проекта и бизнес-задача
Разработка  B2B-платформы для автоматизации оптовых заказов и синхронизации остатков в реальном времени. Проект спроектирован с использованием **AI-assisted разработки** (локальные модели Qwen 2.5 Coder 32B / Qwen 3.6 35B в связке с VS Code + Continue + Ollama).

⚠️ **Примечание по безопасности:** Данный репозиторий представляет собой исключительно архитектурный каркас (скелет) бэкенда. Коммерческая логика, внутренние алгоритмы расчетов и файлы инициализации моделей данных удалены или заменены абстрактными сигнатурами в целях соблюдения NDA. 
---

### 🛠 Технологический стек
* **Backend:** Node.js (Express), REST API, WebSockets (Socket.io).
* **Database Layer:** PostgreSQL, ORM Sequelize.
* **Infrastructure:** Linux Ubuntu, Nginx, PM2.
* **Testing & Security:** Artillery, OWASP ZAP, npm audit.
* **Frontend Component:** React (Vite).

---

### 🏗 Ключевые архитектурные и инженерные решения

#### 1. Многоуровневый контур безопасности (DevSecOps)
Система успешно прошла симуляцию XSS, CSRF и Clickjacking-атаку методом автоматизированного сканирования (Паук + Active Scan) в **OWASP ZAP**.
* **Stateful-авторизация:** Сессии изолированы по организациям (`organizationId`).
* **Изоляция сессий:** Сессионные куки закрыты флагами `httpOnly` и `sameSite: lax`, блокируя кражу сессий через JavaScript-консоль.
* **CSRF-защита:** Интегрирован паттерн *Double Submit Cookie* на базе библиотеки `csrf-csrf`. Токены криптографически привязаны к `sessionID` в PostgreSQL.
* **Динамический CORS:** Валидация входящих доменов (`Origin`) на уровне Express-сервера реализована через регулярные выражения для гибкой поддержки поддоменов клиентов.
* **Защита статики:** Раздача приватных медиа-файлов (прайсы, документы) защищена авторизационным слоем Node.js и делегирована веб-серверу через механизм `X-Accel-Redirect` (Nginx), что исключает нагрузку на RAM процесса Node.js.

#### 2. Оптимизация слоя данных и нагрузки (Highload)
* **Connection Pooling:** Настроена конфигурация пула соединений PostgreSQL (`max: 50`, `min: 5`), что защищает СУБД от исчерпания лимитов при пиковых нагрузках.
* **Event Loop Optimization:** Принудительно отключено SQL-логирование (`logging: false`), так как синхронный вывод логов блокирует основной поток Node.js.
